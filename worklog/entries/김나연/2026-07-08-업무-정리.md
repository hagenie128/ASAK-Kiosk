## 2. 작업 목적

- 관리자 서비스에서 필요한 주요 화면을 Figma로 설계하고 재사용 가능한 UI 컴포넌트로 정리한다.
- 로그인, 주문 현황, 메뉴 관리, 매출 요약, 품절 관리, 결제수단 설정 화면의 상태별 UI를 구성한다.

## 3. 직접 구현 영역

- **관리자 로그인 화면:** Figma에서 관리자 로그인 페이지를 생성하고 로그인 UI 시안을 구성했다.
- **관리자 주문 현황 화면:** 고객 주문 정보를 카드형으로 보여주는 컴포넌트를 만들고, 메뉴 리스트가 길어질 경우를 위한 스크롤바 옵션을 추가했다.
- **관리자 메뉴 관리·등록·수정 화면:** header 컴포넌트 디자인 5가지 유형을 도출하고, 메뉴 리스트 카테고리 선택 여부에 따른 프로퍼티 컴포넌트를 제작했다.
- **관리자 매출 요약 화면:** 매출 상승 추이와 하락 추이 버전으로 사용할 수 있는 요약 컴포넌트를 제작했다.
- **판매 항목 품절 관리 화면:** 메뉴 리스트 컴포넌트를 제작하고, 품절 처리 클릭 시 노출되는 UI 상태를 만들었다.
- **관리자 결제수단 설정 화면:** 카드형 결제수단 선택 UI와 체크박스 토글 유무에 따른 상태 컴포넌트를 제작했다.

## 4. 구현 로직 / 적용한 방식

- 관리자 로그인 화면은 계정 입력과 진입 액션이 명확히 보이도록 간결하게 구성했다.
- 주문 현황은 주문별 정보를 카드 단위로 나누고, 내부 메뉴 리스트 길이에 따라 스크롤 가능한 상태를 고려했다.
- 메뉴 관리 화면은 등록, 수정, 목록 화면에서 필요한 header 유형을 분리해 재사용성을 높였다.
- 매출 요약은 상승과 하락 상태를 별도 컴포넌트 상태로 나눠 데이터 변화 방향이 바로 보이도록 설계했다.
- 품절 관리는 리스트 안에서 판매 가능 상태와 품절 처리 상태가 구분되도록 클릭 후 UI를 따로 만들었다.
- 결제수단 설정은 카드 선택 상태와 체크박스 토글 상태를 분리해 현재 설정값을 명확히 보여주도록 구성했다.

## 6. 발생 이슈

- 이슈 1:
  - 증상: 주문 카드 안의 메뉴 리스트가 길어질 경우 화면 균형이 깨질 수 있음
  - 원인: 주문별 메뉴 개수가 고정되어 있지 않음
  - 해결: 주문 현황 카드에 스크롤바 옵션을 추가해 긴 리스트를 처리할 수 있도록 설계
- 이슈 2:
  - 증상: 메뉴 관리 화면마다 header 역할과 주요 액션이 달라 한 가지 형태로 표현하기 어려움
  - 원인: 등록, 수정, 목록 화면의 목적이 다름
  - 해결: header 컴포넌트 디자인을 5가지 유형으로 분리
- 이슈 3:
  - 증상: 결제수단 설정에서 카드 선택 상태와 체크박스 토글 상태가 동시에 표현되어야 함
  - 원인: 결제수단 종류 선택과 활성 여부가 같은 화면에서 관리됨
  - 해결: 카드 선택 상태와 체크박스 토글 상태를 컴포넌트 프로퍼티로 분리

## 7. 디버깅 기록

- 확인한 로그/에러 메시지: -
- 의심했던 지점: 주문 카드 내부 스크롤, header 유형 분리, 상태별 컴포넌트 표현
- 실제 원인: 관리자 화면은 데이터 길이와 상태 변화가 많아 단일 정적 시안만으로는 부족함
- 다시 같은 문제가 생기면 먼저 볼 파일/명령어: Figma 관리자 화면 컴포넌트 세트

## 8. 이번 작업에서 배운 점

- 관리자 화면은 반복 작업을 빠르게 처리해야 하므로 정보 구조와 상태 구분이 중요하다.
- 카드형 UI는 항목 구분이 쉽지만 내부 정보량이 늘어나는 상황을 함께 고려해야 한다.
- 컴포넌트 프로퍼티를 잘 나누면 화면 상태를 빠르게 바꾸고 일관된 디자인을 유지할 수 있다.
